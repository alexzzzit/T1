# HW_1_SOLUTION.md: Нагрузочное тестирование demo-app-1

## Этап №1: Думаем и проектируем (Метрики)

Опираясь на **модель 4К (Клиент, Канал, Компонент, Кладовая)** из *Занятия №2 (слайд 12)*, мы должны отслеживать метрики на каждом уровне абстракции, чтобы понять, где система «задыхается». Согласно *слайду 17*, для каждого элемента модели предусмотрены свои специфичные метрики:

1. **Клиент (k6 / Виртуальные пользователи):**
   * *RPS (Requests Per Second)* — интенсивность входящего потока ($\lambda$).
   * *Connection time* — время установки соединения.
2. **Канал (NGINX):**
   * *Трафик (Кб/с)* — пропускная способность сети.
   * *Количество соединений* — чтобы понять, не уперлись ли мы в лимиты `worker_connections`.
3. **Компонент (Backend Application):**
   * *Latency* — время обработки запроса (влияет на $W$ в Законе Литтла).
   * *Утилизация ресурсов (CPU, RAM)* — загрузка сервера.
   * *Errors* — число отказов (слайд 16).
4. **Кладовая (База данных):**
   * *TPS/QPS* — реальная пропускная способность БД.
   * *Active / Idle соединения* — критично для понимания, не исчерпан ли пул коннектов.
   * *Утилизация ресурсов (CPU, RAM)* и *Disk I/O*.

**⭐ Что я добавил (под звездочкой):**
Согласно *слайду 19 (Занятие №2)*, главная проблема при скачках нагрузки — это очередь. Я добавил бы метрику **Длина очереди (Queue Length)** и **Время ожидания в очереди (Queue Time)** на уровне Компонента. Это позволит заранее увидеть, что $\lambda$ превышает пропускную способность канала обслуживания, еще до того, как начнутся массовые таймауты.

## Этап №2: Стреляем (Сценарии k6)

Мы используем k6 для эмуляции трафика и проверки поведения системы в разных условиях.

1. **Сценарий "Шторм" (Spike):**
   * *Настройка:* Резкий скачок до 1000 пользователей за 10 секунд.
   * *Цель:* Проверить, как система справится с резким ростом $\lambda$ (интенсивности входящего потока). Ударит ли она в лимиты соединений Канала (NGINX) или уйдет в бесконечную очередь Компонента.
2. **Сценарий "Волна" (Ramp-up):**
   * *Настройка:* Плавный рост от 0 до 500 пользователей за 2 минуты.
   * *Цель:* Найти точку насыщения сервера. Согласно *слайду 21 (Занятие №2)*, "У сервера конечные ресурсы". Мы хотим увидеть момент, когда RPS перестанет расти (плато), а Latency начнет линейно или экспоненциально увеличиваться.
3. **Сценарий "Моя модификация" (Soak Test / Утечка коннектов):**
   * *Настройка:* Средняя нагрузка (100 VUs) на протяжении 15 минут.
   * *В чем фишка:* Я хочу проверить поведение Кладовой (БД) на длительной дистанции. Сценарий сфокусирован на метрике `Active/Idle connections`. Цель — выявить, не происходит ли утечки соединений, когда приложение не закрывает коннекты после обработки 80% POST-запросов (создания заказов).

## Этап №3: Анализируем (Инсайты и Теория)

Анализ дашбордов (Grafana/Prometheus) проводим через призму **Теории массового обслуживания (ТМО)** и **Закона Литтла** (*Занятие №2, слайд 16: $L = \lambda \times W$*).

### Инсайты по сценарию "Шторм":
* **Наблюдение:** При скачке до 1000 VUs NGINX начинает отдавать `502 Bad Gateway`, а Latency на бэкенде улетает в секунды.
* **Обоснование (ТМО):** Интенсивность входящего потока ($\lambda$) многократно превысила пропускную способность канала обслуживания ($\mu$). Согласно *слайду 19*, запросы начинают скапливаться в очереди. Так как у сервера конечные ресурсы (*слайд 21*), пул потоков/воркеров исчерпывается. NGINX не может дождаться ответа и разрывает соединение (504/502).

### Инсайты по сценарию "Волна":
* **Наблюдение:** При достижении ~250 VUs график RPS выходит на плато. Дальнейший рост VUs не увеличивает RPS, но резко увеличивает Latency (p95). CPU БД упирается в 100%.
* **Обоснование:** Мы достигли физического предела пропускной способности Кладовой (БД) на запись (так как 80% трафика — это POST-запросы). БД стала "узким местом" (ботлнеком). Согласно *Занятию №2 (слайд 28)*, вертикальное масштабирование имеет предел, ограниченный железом.

### Инсайты по сценарию "Моя модификация":
* **Наблюдение:** К 10-й минуте график `Active connections` в БД ползет вверх, а `Idle connections` падает до нуля. Latency начинает деградировать.
* **Обоснование:** Обнаружена проблема на уровне интеграции Компонента и Кладовой. Приложение открывает новые соединения для обработки POST-запросов, но не возвращает их в пул (или пул настроен некорректно). БД исчерпывает свой лимит `max_connections`.

## Этап №4: Решения (Как справится с нагрузкой)

Опираясь на пройденный материал, предлагаю следующие архитектурные и инфраструктурные решения:

### 1. Решение проблемы "Шторма" и очереди (Канал и Компонент)
* **Rate Limiter (Ограничение запросов):** Согласно *Занятию №9 (слайд 22)*, мы должны защищать емкость системы (Anti-DDoS). Настроим NGINX или API Gateway так, чтобы при превышении безопасного RPS система отдавала `429 Too Many Requests`, а не уходила в каскадный отказ и не "вешала" БД.
* **Паттерны отказоустойчивости (Timeout, Retry, Circuit Breaker):**
  * *Timeout:* Жестко ограничим время ожидания ответа от БД, чтобы не держать потоки App в висящем состоянии (*Занятие №9, слайд 16*).
  * *Circuit Breaker:* Если БД "лежит" или отвечает слишком долго, автоматический выключатель прекратит ходить в нее, предотвращая каскадный сбой (*Занятие №9, слайд 20*).
  * *Retry с Exponential Backoff:* Для идемпотентных запросов добавим ретраи с увеличением задержки, чтобы не устроить "retry storm" (*Занятие №9, слайд 18-19*).

### 2. Решение проблемы плато RPS и нагрузки на БД (Кладовая)
* **Кэширование (Sidecar-кэш):** Согласно *Занятию №8 (слайд 7)*, кэширование позволяет хранить данные как можно ближе к месту использования. 20% нагрузки — это GET-запросы. Внедрив Redis (Sidecar-кэш) и используя паттерн *Cache-aside* или *Write-through* (*слайд 9*), мы снимем до 80% read-нагрузки с БД (принцип Парето 80/20, *слайд 7*). Это снизит $W$ (Latency) и увеличит общий Throughput.
* **Горизонтальное масштабирование (Бэкенд):** Согласно *Занятию №2 (слайд 28)*, горизонтальное масштабирование (добавление "каналов обслуживания") теоретически не ограничено и повышает надежность. Мы должны сделать Backend stateless (согласно принципам REST, *Занятие №6, слайд 4* и 12-Factor App, *Занятие №4, слайд 26*) и добавить Load Balancer.
* **Алгоритм балансировки:** Согласно *Занятию №7 (слайд 27)*, для нашего случая (разнородные по времени выполнения запросы: долгие POST и быстрые GET) алгоритм *Round Robin* не подойдет. Нужно использовать **Leastconn** (отправлять запрос на сервер с наименьшим числом активных соединений) или **Least time**, чтобы равномерно распределять нагрузку между репликами бэкенда.

### 3. Решение проблемы утечки коннектов (Интеграция)
* **Оптимизация пула соединений:** Настроить `max_idle_connections` и `connection_max_lifetime` на стороне приложения.
* **Асинхронная интеграция (Message Queue):** Если часть POST-запросов (например, создание заказов) не требует мгновенного ответа пользователю, мы можем перейти от синхронной интеграции к асинхронной через Message Queue (RabbitMQ/Kafka). Согласно *Занятию №5 (слайд 20)*, это создаст "естественный буфер", пиковая нагрузка будет буферизироваться в брокере, а БД будет обрабатывать сообщения со своей комфортной скоростью (TPS), без очередей и 502-х ошибок.

## Диаграмма архитектуры (C4 Model)

Ниже представлен код для Structurizr DSL, описывающий архитектуру `demo-app-1` с учетом предложенных решений (добавлены Load Balancer, Redis Cache и Message Queue для разгрузки БД).

```dsl
workspace {
    model {
        user = person "Пользователь" "Виртуальный пользователь (k6), создающий и просматривающий заказы"
        
        demoApp = softwareSystem "Demo App 1" "Веб-приложение для управления заказами" {
            
            # Клиент (Модель 4К)
            client = container "HTTP Client" "k6 / Браузер" "Генерирует нагрузку (RPS)"
            
            # Канал (Модель 4К)
            nginx = container "NGINX" "Reverse Proxy / Load Balancer" "Терминация соединений, Rate Limiting, балансировка (алгоритм Leastconn)"
            
            # Компонент (Модель 4К)
            backend = container "Backend API" "Go / Python / Node.js" "Бизнес-логика, обработка POST/GET запросов. Stateless."
            
            # Кладовая (Модель 4К) - Оптимизированная
            db = container "PostgreSQL" "Реляционная БД" "Хранение заказов. Оптимизирован пул коннектов."
            cache = container "Redis" "In-Memory KV" "Sidecar-кэш для GET-запросов (паттерн Cache-aside). Снижает нагрузку на БД."
            queue = container "RabbitMQ" "Message Queue" "Буферизация тяжелых POST-запросов (асинхронная интеграция)."
            worker = container "Background Worker" "Go / Python" "Асинхронно разбирает очередь и пишет в БД с комфортным TPS."
        }
        
        # Отношения
        user -> client "Инициирует запросы" "HTTP/HTTPS"
        client -> nginx "Создает соединения" "HTTP"
        
        nginx -> backend "Проксирует запросы" "HTTP/gRPC (Round Robin / Leastconn)"
        nginx ..> client "Отдает 429 при превышении лимита" "Rate Limiting"
        
        backend -> cache "Читает кэш (GET)" "TCP"
        backend -> db "Пишет/Читает напрямую (если нет в кэше)" "SQL (TCP)"
        backend -> queue "Отправляет задачи на создание заказа" "AMQP (Асинхронно)"
        
        worker -> queue "Забирает задачи" "AMQP"
        worker -> db "Сохраняет заказ в БД" "SQL (TCP)"
    }

    views {
        container demoApp "Containers" "Контейнерная диаграмма Demo App 1 с учетом решений по масштабированию и кэшированию" {
            include *
            autoLayout lr
        }
    }
}
```
