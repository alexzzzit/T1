# HW_3_SOLUTION.md: High Level Design (Бухгалтерская система колледжа)

## Контекст
- **Домен**: Бухгалтерский учет и управление расходами колледжа.
- **Ограничения**: Отсутствие веб-интерфейса (только Desktop и Mobile), минимальный бюджет, строгая сертификация ФСТЭК, неизменяемый Audit Trail.
- **Масштаб**: ~1000 пользователей, низкий RPS (~20 в пике), критичные требования к консистентности и безопасности.

---

## Part 1. Модули и интеграции

### 1.1. Декомпозиция доменов (DDD)
Опираясь на **Занятие №4 (слайд 14, Стратегический DDD)** и **Занятие №5 (Event Storming)**, мы выделяем ограниченные контексты (Bounded Contexts) на основе бизнес-событий и языковых границ:
1. **Identity & Security (Идентификация и Аудит)**: Управление доступом, 2FA, ведение неизменяемого лога аудита. Отдельный контекст, так как аудит имеет собственную семантику (append-only, хэширование по ГОСТ).
2. **Core Finance & Workflow (Финансы и Маршруты)**: Проводки, бюджеты, движок бизнес-процессов. Единый контекст, так как финансы и маршруты согласования тесно связаны (**Connascence of Execution**, Занятие №4 слайд 10 — порядок шагов критичен).
3. **Document Management (Документооборот)**: Хранение сканов, чеков, актов. Отдельный контекст, так как работа с бинарными объектами имеет иную природу (**Connascence of Type**, Занятие №4 слайд 6).

### 1.2. Обоснование архитектурного стиля
Согласно **Занятию №4 (слайды 16-18, связь «Архитектура» и «Фаза продукта»)**, выбор стиля зависит от фазы продукта и доменной зрелости.

**Анализ альтернатив (Занятие №3, слайд 20):**
| Критерий | Модульный монолит | Микросервисы |
|---|---|---|
| **Деплой** | Один артефакт — достаточно для 1000 пользователей | Избыточно: N деплоев, оркестрация |
| **Коммуникация** | Локальные вызовы (быстро, надежно) | Сетевые вызовы (задержки, точки отказа) |
| **Данные** | Общая БД, разделенные схемы — критично для ACID | Database-per-service усложняет транзакции |
| **Стоимость изменений** | Низкая — рефакторинг в рамках одного кода | Высокая — координация нескольких сервисов |

**Решение**: **Модульный монолит** с физическим выделением тяжелого Document Service в отдельный процесс (разделенный монолит).
**Обоснование (Занятие №4, слайд 18)**:
- На старте домен неочевиден («некорректно разбив границы в начале — дальше заплатим дважды за рефакторинг»).
- 1000 пользователей не требуют горизонтального масштабирования отдельных модулей.
- Принцип «жить по карману»: микросервисы дороже не только по инфраструктуре, но и по инженерному времени.

### 1.3. Интеграции между сервисами
Согласно **Занятию №5 (слайды 18-22)**, выбор синхронного или асинхронного взаимодействия зависит от бизнес-требований к задержке и надежности.

**Синхронные интеграции (REST/gRPC):**
- **Клиенты → API Gateway → Core Backend**: Пользователю нужен немедленный отклик при сохранении документа (сценарий «пользователь открыл корзину», Занятие №5 слайд 22).
- **Core Backend → Auth & Audit (проверка прав)**: **Fail Fast** (Занятие №9 слайд 21). Права проверяем синхронно. Если IdP недоступен — отказываем в доступе.

**Асинхронные интеграции (Message Queue):**
- **Core Backend → Document Service (загрузка сканов)**: Сценарий «гарантированная доставка документов» (Занятие №5 слайд 22). Загрузка + антивирус + OCR — долгая операция. Используем очередь, чтобы не занимать поток Backend. Producer работает, даже если Document Service лежит.
- **Core Backend → Notification Service**: Уведомления — не критичный путь, могут быть доставлены с задержкой. Eventual consistency приемлема.

**Обоснование выбора брокера (Занятие №5 слайд 27)**:
Выбираем **RabbitMQ** (Message Queue), а не Kafka (Message Broker). У нас нет потоковой аналитики, нужны именно очереди задач для конкретных consumer-ов. RabbitMQ проще в эксплуатации и имеет меньший оверхед, что соответствует требованию «минимальный бюджет».

## Part 2. Выбор баз данных

Для выбора СУБД применяем **Алгоритм из 11 вопросов** (Занятие №6 слайд 29).

### 2.1. Core Finance & Workflow DB
1. **Что за данные?** Финансовые проводки, счета, бюджеты.
2. **Формат?** Строго структурированный, реляционный.
3. **Отношения (ERD)?** Сложные, жесткие связи (Счет → Проводка → Документ).
4. **Сценарии?** OLTP, строгие транзакции (правило двойной записи), блокировки строк.
11. **Сертификация?** **Postgres Professional** имеет сертификат ФСТЭК.

**Решение**: **PostgreSQL**.
**Обоснование**: Требование ACID (Занятие №6 слайд 12) критично для финансов:
- **Atomicity**: Проводка либо выполняется полностью, либо не выполняется.
- **Consistency**: Бизнес-правила обеспечиваются CHECK-ограничениями.
- **Isolation**: Два бухгалтера не должны провести один документ дважды.
NoSQL (BASE, eventual consistency) **не подходит** — финансовая отчетность требует строгой согласованности.

### 2.2. Auth & Audit DB
**Решение**: **PostgreSQL** (в той же инсталляции).
**Особенность — Партиционирование по датам** (Занятие №6 слайд 27, логическое разделение):
- Таблица разбивается на партиции по месяцам.
- Старые партиции переводятся в read-only — физическая защита от модификации (требование ИБ).
- Ускоряются запросы аудиторов (scanning только нужных партиций).

### 2.3. Document Storage
**Решение**: **Object Storage (MinIO)**.
**Обоснование**: Хранить бинарники в PostgreSQL — антипаттерн (раздувание БД). MinIO предоставляет S3-совместимый API, поддерживает On-Premise развертывание и имеет сертификаты ФСТЭК.

### 2.4. Репликация и шардирование
- **Репликация**: **Active-Passive (Primary + Hot Standby)** через Streaming Replication и Patroni + etcd (Занятие №9 слайды 8-9). **RPO = 0** (синхронная репликация гарантирует, что ни одна транзакция не потеряется).
- **Шардирование**: **Не требуется** (Занятие №6 слайд 27). Физическое разделение данных добавит операционной сложности, что противоречит требованию «минимальный бюджет» при текущей нагрузке (~20 RPS).
- **Кэширование**: **Redis Cluster** для кэша справочников. Паттерн **Cache-aside** (Занятие №8 слайд 9), TTL 5-10 минут (**Temporal Locality**), вытеснение **LRU** (Занятие №8 слайд 10).

## Part 3. Дополнительные компоненты

### 3.1. Обязательные компоненты (MUST)

| Компонент | Обоснование (по лекциям) | Реализация |
|---|---|---|
| **Load Balancer / Reverse Proxy** | Точка входа, терминация TLS/ГОСТ. Без него невозможно горизонтальное масштабирование (Занятие №9 слайд 14). | **NGINX** |
| **Cache (In-Memory)** | Снижение нагрузки на БД для «горячих» данных (справочники). Паттерн Cache-aside, принцип Парето 80/20 (Занятие №8 слайд 7). | **Redis** |
| **Message Broker** | Буферизация тяжелых задач. Producer работает, даже если consumer лежит (Занятие №5 слайд 20). | **RabbitMQ** |
| **Observability** | 4 столпа: метрики, логи, трейсы, профили. Критично для расследования инцидентов ИБ (Занятие №7 слайд 23). | **Prometheus + Grafana + Loki** |
| **Backup & DR** | Гарантия восстановления (RPO=0 через WAL-G). Требование ФСТЭК (Занятие №9 слайд 17). | **WAL-G** + Snapshots |
| **IdP (Identity Provider)** | Централизованная аутентификация, 2FA (RuToken/ГОСТ) (Занятие №9 слайд 27). | **Keycloak** |

### 3.2. Рекомендуемые компоненты (SHOULD)

| Компонент | Обоснование | Реализация |
|---|---|---|
| **WAF** | Защита от OWASP Top 10. Требуется для сертификации ФСТЭК (Занятие №9 слайд 26). | **NGINX WAF** (ModSecurity) |
| **API Gateway** | Централизованный Rate Limiting (Anti-DDoS) и маршрутизация (Занятие №5 слайд 28). | **Kong** или **Tyk** |
| **CI/CD** | Автоматизация деплоя. Снижает риск человеческого фактора (Занятие №4 слайд 26, 12-Factor App). | **GitLab CI** |
| **Service Mesh** | **Отклоняем на данном этапе**. Внедрение Istio для 2-3 сервисов нарушит принцип минимального бюджета (Занятие №4 слайд 34). | N/A |

## Диаграмма архитектуры (C4 Model)

Ниже представлен код для Structurizr DSL. Сохраните его в файл `workspace.dsl` и откройте на [app.structurizr.com](https://app.structurizr.com/).

```dsl
workspace {

    model {
        # Акторы
        buhgalter = person "Бухгалтер" "Вводит документы, проводит платежи"
        sotrudnik = person "Сотрудник" "Подает заявки на расходы"
        zav_kaf = person "Зав. кафедрой" "Утверждает заявки удаленно"
        auditor = person "Аудитор ИБ" "Проверяет журналы аудита"

        # Внешние системы
        bank = softwareSystem "Банк-клиент" "Проведение платежей"
        nalog = softwareSystem "ФНС" "Электронная отчетность"
        ldap = softwareSystem "Корпоративный LDAP" "Хранилище учетных записей"

        # Наша система
        accountingSystem = softwareSystem "Бухгалтерская система колледжа" "Учет расходов, workflow, неизменяемый аудит. ФСТЭК." {
            
            # Клиенты
            desktopApp = container "Desktop Client" "Qt/C#" "Основной интерфейс в LAN. Поддержка оффлайн."
            mobileApp = container "Mobile App" "Flutter" "Согласование заявок через VPN."

            # Канал / Точка входа
            apiGateway = container "API Gateway" "NGINX + WAF" "Терминация TLS, Rate Limiting, маршрутизация."

            # Компоненты
            backend = container "Core Backend" "Go/Java" "Модульный монолит: финансы, Workflow Engine."
            authService = container "Auth & Audit" "Go" "Аутентификация (LDAP), запись в неизменяемый журнал."
            docService = container "Document Service" "Go" "Обработка файлов: антивирус, OCR, PDF."

            # Кладовые
            postgresPrimary = container "PostgreSQL Primary" "Postgres Pro" "Основная БД (ACID). Single-Leader."
            postgresReplica = container "PostgreSQL Standby" "Postgres Pro" "Hot Standby для чтения и тяжелых отчетов."
            auditDb = container "Audit Database" "Postgres Pro" "Append-only логи. Партиционирование по датам."
            redis = container "Redis" "Redis" "Кэш справочников и сессий (Cache-aside, LRU)."
            minio = container "MinIO" "S3" "Хранилище сканов и бэкапов."

            # Брокер
            rabbitmq = container "RabbitMQ" "RabbitMQ" "Очередь задач для Document Service."
        }
    }

    views {
        # System Context
        systemContext accountingSystem "SystemContext" "Контекст системы" {
            include *
            rel buhgalter, accountingSystem "Вводит документы", "Desktop, LAN"
            rel sotrudnik, accountingSystem "Подает заявки", "Desktop/Mobile"
            rel zav_kaf, accountingSystem "Утверждает", "Mobile, VPN"
            rel auditor, accountingSystem "Проверяет логи", "Desktop"
            rel accountingSystem, bank "Платежные поручения", "HTTPS/API"
            rel accountingSystem, nalog "Отчетность", "HTTPS/API"
            rel accountingSystem, ldap "Аутентификация", "LDAPS"
            autoLayout lr
        }

        # Container Diagram
        container accountingSystem "Containers" "Технологические компоненты" {
            include *
            
            # Клиенты -> Шлюз
            rel buhgalter, desktopApp "Использует", "LAN"
            rel sotrudnik, mobileApp "Использует", "VPN"
            rel zav_kaf, mobileApp "Утверждает", "VPN"
            rel auditor, desktopApp "Использует", "LAN"
            
            rel desktopApp, apiGateway "HTTPS", "REST/JSON"
            rel mobileApp, apiGateway "HTTPS", "REST/JSON"
            
            # Шлюз -> Бэкенд
            rel apiGateway, backend "gRPC", "Внутренний API"
            rel apiGateway, authService "gRPC", "Проверка токенов"
            
            # Бэкенд -> Хранилища и Брокер
            rel backend, postgresPrimary "SQL", "Основная БД (ACID)"
            rel backend, redis "TCP", "Кэш справочников"
            rel backend, rabbitmq "AMQP", "Асинхронные задачи"
            
            # Аудит и Документы
            rel authService, auditDb "SQL", "Append-only журнал"
            rel authService, ldap "LDAPS", "Аутентификация"
            
            rel backend, docService "gRPC", "Синхронные вызовы"
            rel docService, rabbitmq "AMQP", "Получает задачи"
            rel docService, minio "S3 API", "Хранение файлов"
            
            # Репликация
            rel postgresPrimary, postgresReplica "Streaming Replication", "Синхронная репликация"
            
            autoLayout lr
        }
    }
}
```
