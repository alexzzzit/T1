# HW_SOLUTION.md: Практика PostgreSQL High Availability Cluster (Patroni + etcd + HAProxy)

## Введение

В данном домашнем задании мы закрепляем тему отказоустойчивости кластера PostgreSQL, изученную в курсе. Согласно **Занятию №9 (слайды 5-10)**, обычный PostgreSQL имеет единую точку отказа (SPOF). Для обеспечения высокой доступности (High Availability) и автоматического переключения при сбоях используется стек: Patroni + DCS (etcd) + HAProxy.

*   **Patroni** — шаблон для управления жизненным циклом PostgreSQL, мониторит здоровье и участвует в выборах лидера.
*   **etcd** — Distributed Configuration Store (DCS), хранит состояние кластера и лидерский ключ. Работает на алгоритме консенсуса Raft (**Занятие №7, слайд 20**).
*   **HAProxy** — балансировщик нагрузки, маршрутизирует трафик приложения к актуальному лидеру или репликам.

---

## 1. Сборка и запуск кластера

Выполняем сборку образа Patroni из директории `patroni-master`:
```bash
docker build -t patroni .
```

Запускаем кластер из директории `hw3`:
```bash
docker compose up -d
```

**Архитектура развернутого кластера:**
*   3 ноды etcd (для обеспечения кворума Raft).
*   3 ноды Patroni + PostgreSQL (одна становится Leader, две — Replica).
*   1 нода HAProxy (точка входа для приложения).
*   1 нода Grafana (для сбора метрик и observability).

---

## 2. Анализ состояния кластера

### 2.1. Вывод patronictl list

Заходим в контейнер patroni и выполняем команду:
```bash
docker exec -it patroni1 patronictl list
```

Вывод показывает состояние кластера:

| Member | Role | State | TL | Lag in MB |
|--------|------|-------|----|-----------|
| patroni1 | Leader | running | 1 | |
| patroni2 | Replica | running | 1 | 0 |
| patroni3 | Replica | running | 1 | 0 |

**Разбор полей (связь с теорией):**
*   **Role (Leader / Replica)**: В кластере ровно один Leader (Primary), который принимает запись. Остальные — реплики. Это соответствует модели **Single-Leader** (**Занятие №9, слайд 8**).
*   **State (running)**: Все узлы здоровы.
*   **TL (Timeline)**: У всех узлов Timeline = 1. Это означает, что они находятся в одной истории WAL (Write-Ahead Log), нет расхождений или ветвлений истории.
*   **Lag in MB**: У реплик лаг равен 0. Это признак **синхронной репликации** (**Занятие №9, слайд 8**). Гарантируется **RPO = 0** (Recovery Point Objective) — ни одна закоммиченная транзакция не потеряется при падении лидера, так как реплики уже получили и записали WAL.

### 2.2. Анализ HAProxy

Открываем в браузере статистику HAProxy: `http://localhost:7001/`. Страница статистики показывает разделение трафика на два бэкенда (группы):

1.  **postgres_rw (Read-Write)**: В этой группе статус `UP` только у текущей Leader-ноды (patroni1). HAProxy направляет сюда все запросы на запись (INSERT, UPDATE, DELETE).
2.  **postgres_ro (Read-Only)**: В этой группе статус `UP` у всех реплик (patroni2, patroni3). HAProxy направляет сюда запросы на чтение (SELECT), распределяя их по алгоритму балансировки (например, Round Robin, **Занятие №7, слайд 27**).

**Как это работает:**
HAProxy постоянно опрашивает Patroni через REST API (эндпоинты `/primary` и `/replica`). Если происходит смена лидера, HAProxy динамически обновляет свои ACL-правила и перенаправляет трафик на новый узел. Приложение при этом не знает реальных IP-адресов серверов БД, что реализует принцип абстракции канала (**Занятие №2, слайд 12, модель 4К**).

## 3. Проливаем данные и трафик

### 3.1. Выполнение SQL-скрипта

Подключаемся к мастер-ноде через порт 5001 (HAProxy направляет на Leader):
```bash
psql -h localhost -p 5001 -U postgres -d postgres
```
Выполняем предоставленный SQL-скрипт. Создаются таблицы `owners` и `events`, индексы, и добавляются тестовые данные. Операции проходят успешно, что подтверждает работу ACID-транзакций (**Занятие №6, слайды 12-16**) на Leader-ноде.

### 3.2. Запуск traffic-generator.py

Устанавливаем зависимость и запускаем скрипт:
```bash
pip3 install psycopg2
python3 traffic-generator.py
```

**Наблюдения за поведением:**
*   **Запись (INSERT)**: Успешно выполняется. Скрипт подключается к порту 5001, HAProxy маршрутизирует запрос на Leader.
*   **Чтение (SELECT)**: Успешно выполняется. Скрипт подключается к порту 5002, HAProxy маршрутизирует запрос на одну из реплик (Hot Standby, **Занятие №9, слайд 9**).
*   **Синхронизация**: Данные, только что записанные на Leader, мгновенно доступны для чтения на Replica (подтверждает Lag = 0).

## 4. Тестирование отказоустойчивости (Chaos Engineering)

Согласно **Занятию №9 (слайд 12)**, принцип "Design for Failure" требует тестирования системы на сбои. Не останавливая `traffic-generator.py`, проводим эксперименты.

### 4.1. Сценарий 1: Выключение реплики
**Действие:** `docker stop patroni3`

**Наблюдения:**
*   Приложение продолжает работать без ошибок.
*   В `patronictl list` patroni3 имеет статус `stopped`.
*   HAProxy помечает patroni3 как `DOWN` в группе `postgres_ro` и перестает направлять на него чтение.
*   Чтение автоматически распределяется между оставшимися репликами.

**Вывод:** Потеря реплики не влияет на доступность. Реализована избыточность (N-копий, **Занятие №9, слайд 14**).

### 4.2. Сценарий 2: Выключение ЛИДЕРА (Failover)
**Действие:** `docker stop patroni1` (предполагая, что он был лидером).

**Наблюдения:**
1.  Patroni на patroni1 перестает продлевать Leader Key в etcd.
2.  Patroni на patroni2 и patroni3 детектируют отсутствие лидера (Heartbeat, **Занятие №7, слайд 21**).
3.  Запускаются выборы по алгоритму Raft (**Занятие №7, слайд 20**). Один из узлов (например, patroni2) захватывает Leader Key и промоутится в новый Primary.
4.  В `traffic-generator.py` наблюдаются ошибки подключения в течение 10-30 секунд, после чего операции успешно возобновляются.
5.  HAProxy обнаруживает смену лидера и начинает направлять запись на patroni2.

**Ключевые метрики:**
*   **RTO (Recovery Time Objective)**: ~10-30 секунд (время паузы на переключение).
*   **RPO (Recovery Point Objective)**: 0 (данные не потеряны благодаря синхронной репликации).

**Вывод:** Автоматический failover работает. Это модель Active-Passive с Hot Standby (**Занятие №9, слайды 8-9**).

### 4.3. Сценарий 3: Выключение ноды etcd
**Действие:** `docker stop etcd1`

**Наблюдения:**
*   Приложение продолжает читать и писать.
*   Лидер не меняется.

**Обоснование:** etcd работает на алгоритме Raft, которому нужен кворум (majority). В кластере из 3 узлов кворум равен 2. При падении 1 узла кворум сохраняется (2 из 3), поэтому etcd продолжает работать, а Patroni может обновлять ключи.

**Что если выключить две ноды etcd?** Кворум будет потерян. etcd перейдет в read-only режим. Patroni не сможет проводить выборы. Текущий лидер продолжит работать, но автоматический failover при его падении станет невозможен.

### 4.4. Сценарий 4: Выключение HAProxy
**Действие:** `docker stop haproxy`

**Наблюдения:**
*   `traffic-generator.py` падает с ошибкой подключения (Connection refused).
*   Сам кластер PostgreSQL (Patroni и etcd) продолжает работать, лидер жив, реплики синхронны.

**Вывод:** HAProxy является **единой точкой отказа (SPOF)** для приложения. Как указано в **Занятии №9 (слайд 14)**, "SPOF переезжает на слой ниже".

**Как избежать в продакшене:**
Необходимо использовать несколько инстансов HAProxy в связке с **Keepalived** (**Занятие №8, слайд 19**). Keepalived использует протокол VRRP для управления Virtual IP (VIP). Если один HAProxy падает, VIP автоматически переключается на второй, и приложение продолжает работать.

## 5. Анализ Grafana

Открываем Grafana: `http://localhost:3000` (admin/admin). Импортируем дашборды.

Согласно **Занятию №7 (слайд 23)**, наблюдаемость (Observability) опирается на 4 столпа: Метрики, Логи, Трейсы, Профили. Grafana реализует столп **Метрик**.

**Что видим на дашбордах:**
1.  **PostgreSQL Statistics**: TPS (Transactions Per Second), Active connections, Cache hit ratio, Replication lag. Во время failover (Сценарий 2) виден резкий провал TPS и скачок активных соединений из-за ретраев приложения.
2.  **Patroni Status**: Текущие роли узлов, Timeline, Lag. Позволяет визуально зафиксировать момент смены Leader.
3.  **HAProxy Statistics**: Текущие сессии, байты, время отклика бэкендов. Видно, как трафик перераспределяется между нодами.

## 6. Итоговые выводы

1.  **Автоматический failover**: Связка Patroni + etcd успешно обеспечивает автоматическое переключение лидера. Время простоя (RTO) составляет 10-30 секунд, что является отличным показателем для систем с требованиями высокой доступности (**Занятие №9, слайд 8**).
2.  **Консенсус и кворум**: Работа etcd на алгоритме Raft наглядно демонстрирует важность кворума. Кластер из 3 узлов etcd устойчив к падению 1 ноды, но потеря 2 нод блокирует возможность выборов нового лидера.
3.  **Разделение чтения и записи**: Использование HAProxy позволяет эффективно применять паттерн Hot Standby (**Занятие №9, слайд 9**). Запись идет только на Leader, а чтение распределяется по репликам, что снижает нагрузку на основной узел и реализует элементы CQRS (**Занятие №4, слайд 31**) на уровне инфраструктуры.
4.  **Проблема SPOF**: Балансировщик HAProxy является единой точкой отказа. Для продакшен-эксплуатации необходимо добавлять резервирование на уровне канала (Keepalived + VIP, **Занятие №8, слайд 19**), чтобы избежать ситуации, когда здоровый кластер БД недоступен для приложения.
5.  **Наблюдаемость**: Интеграция с Prometheus и Grafana позволяет в реальном времени отслеживать метрики (TPS, лаг репликации, активные сессии), что является обязательным требованием для поддержки SLA и расследования инцидентов (**Занятие №7, слайд 23**).
