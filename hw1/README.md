# Системный дизайн: Бухгалтерская система колледжа "Show Me The Money!"

## Общая информация
**Контекст**: Россия, 2026 год. Сертификация ФСТЭК.

**Вводные**:
- Пользователи: 1000+ (локально + удаленно)
- Ограничения: Нет веб-приложения, минимальные затраты на ПО
- Требования: Workflow с возможностью кастомизации кафедрами, полный audit trail

---

## 1. Отсев требований

### Анализ противоречий (по методологии из Занятия №1)

**"Нет веб-приложения" vs "Удаленные пользователи"**
- Решение: Desktop-клиент в LAN + VPN (ViPNet) для удаленного доступа
- Обоснование: Принцип 4К (Занятие №2) - разделение Client и Channel

**"Минимальные затраты" vs "Сертификация ИБ"**
- Решение: Open Source + российские форки (Astra Linux, Postgres Pro)
- Соответствует: требованиям импортозамещения (Занятие №1, ограничения #16)

**Audit trail**
- Решение: Append-only таблицы + хэширование по ГОСТ
- Требование: неизменяемость логов (Занятие №9, паттерны наблюдаемости)

**Итог отсева**:
- Включено: Desktop, Mobile для согласований, VPN, строгий аудит
- Исключено: Веб-интерфейс, SaaS, публичный доступ

## 4. Нефункциональные требования

### Производительность (Занятие №2, ТМО)

**Метрики**:
- Отклик Desktop: < 1 сек (сохранение документа)
- Отчет за год: < 5 сек (материализованные представления)
- Одновременных пользователей: 800 в пике

**Закон Литтла**: L = λ × W
- Целевая задержка (W): минимизация через кэширование

### Масштабируемость (Занятие №2, №4)

**Горизонтальное масштабирование**:
- Stateless backend (12-Factor App #6)
- Балансировщик нагрузки (Round Robin/Leastconn, Занятие №7)

**Вертикальное масштабирование**:
- PostgreSQL: увеличение RAM/CPU при росте DAU

### Отказоустойчивость (Занятие №8, №9)

**HA PostgreSQL**:
- Active-Passive (Primary + Hot Standby)
- Patroni + etcd для автоматического failover
- RPO = 0 (синхронная репликация)
- RTO < 2 часа

**Паттерны надежности**:
- Timeout: 30 сек для внешних вызовов
- Retry: Exponential Backoff (base_delay=0.5s, max=10s)
- Circuit Breaker: для интеграций с банком
- Fallback: кэш при недоступности БД

### Безопасность (Занятие №9)

**Требования ФСТЭК**:
- Класс защиты: И1/И2
- 2FA: RuToken/ГОСТ-сертификаты
- RBAC: разграничение на уровне строк
- Шифрование: TLS с ГОСТ-шифрами
- WAF: защита веб-шлюза (для Mobile API)

**Соответствие законам**:
- 152-ФЗ: ПДн в РФ
- Приказ ФСТЭК №17/21

### Наблюдаемость (Занятие №7, №8)

**4 столпа observability**:
1. Метрики: Prometheus + Grafana (RPS, latency, errors)
2. Логи: ELK/Loki (централизованная агрегация)
3. Трейсы: Jaeger (distributed tracing)
4. Профили: Pyroscope (continuous profiling)

**Audit**: отдельный SIEM-контур

## 6. Расчет нагрузки

### Методология (Занятие №1, №2)

**Исходные данные**:
- DAU: 600 пользователей
- Действий в день: 30 (вход, просмотр, создание, согласование)
- Пиковый коэффициент: 10 (закрытие периода)

### Расчет RPS (формула из Занятия №1)

**Средний RPS**:
```
RPS_avg = (DAU × действий) / 86400
RPS_avg = (600 × 30) / 86400 = 0.2 RPS
```

**Пиковый RPS** (утренний час, 40% трафика):
```
RPS_peak = (600 × 30 × 0.4) / 3600 = 2 RPS
```

**Экстремальный пик** (коэффициент 10):
```
RPS_extreme = 2 × 10 = 20 RPS
```

**Вывод**: Система не HighLoad, узкие места - сложные SQL-запросы и блокировки БД (Занятие №6, ACID)

### Нагрузка на хранилище

**Аудит** (append-only):
```
1000 users × 50 действий × 250 дней = 12.5M записей/год
12.5M × 1KB = 12.5 GB/год (чистые данные)
С индексами и хэшами: ~50 GB/год
```

**Файлы** (сканы чеков):
```
5000 документов/мес × 2MB = 10 GB/мес → 120 GB/год
```

### Ресурсы (capacity planning, Занятие №8)

| Компонент | vCPU | RAM | Storage | Обоснование |
|-----------|------|-----|---------|-------------|
| App Server | 4 | 8GB | 50GB SSD | 2 ноды (Active-Passive), 20 RPS |
| DB Primary | 8 | 32GB | 500GB SSD | Hot data в RAM, write-intensive |
| DB Replica | 4 | 16GB | 500GB SSD | Read queries, отчеты |
| File Storage | 2 | 4GB | 2TB HDD | MinIO/S3 для сканов |

**Кэширование** (Занятие №8):
- Redis для сессий и токенов
- Стратегия: Cache-aside
- TTL: 5-10 мин (Temporal Locality)
- Вытеснение: LRU

## Архитектурные решения (ADR)

### ADR-001: Выбор архитектуры (Занятие №5)

**Контекст**: Ограничение "нет веб-приложения" + минимальный бюджет

**Решение**: Модульный монолит (Занятие №3, №4)
- Desktop-клиент для бухгалтерии
- Mobile для согласований
- Единый backend

**Отклонено**: Микросервисы (over-engineering для 1000 users)

### ADR-002: СУБД (Занятие №6)

**Контекст**: Требование ACID + сертификация ФСТЭК

**Решение**: PostgreSQL (сертифицированный форк)
- Репликация: streaming replication
- Партиционирование: по датам (audit trail)

**Отклонено**: NoSQL (не подходит для финансовой отчетности)

### ADR-003: Интеграции (Занятие №5)

**Контекст**: Интеграция с банком и налоговой

**Решение**: 
- Синхронные: REST/HTTPS для запросов баланса
- Асинхронные: Очереди для отчетов (гарантированная доставка)

**Паттерны**: Timeout + Retry + Circuit Breaker
